# 15 — Технический стек (рекомендация)

> Отвечает на прямой вопрос клиента: «может, дешевле отдельно мобильное приложение + сайт на WordPress?» и обосновывает выбор под рост в Модель 2.

## Рекомендация (кратко)
**Next.js (App Router) + TypeScript + Tailwind + next-intl + headless-CMS + деплой Vercel.** Лид → CRM + WhatsApp. Главные выгоды — **сейчас** (скорость мобайла, мультиязык/hreflang без плагинного ада, кастомная воронка/конфигуратор, контроль SEO/GEO); **общий бэкенд под будущее приложение** (Модель 2) — бонус: переиспользуем данные и API, минимизируя переписывание.

## Почему так
| Требование проекта | Как закрывает стек |
|---|---|
| Премиум-визуал, быстрый мобайл (дыра Boombastic) | Next.js SSR/SSG, оптимизация изображений/видео, edge-CDN Vercel |
| SEO + GEO (AI-поиск) | SSR (контент виден краулерам/AI), schema (JSON-LD), `hreflang`, sitemap, контроль robots.txt/llms.txt |
| Мультиязык EN/DE/RU + рост | `next-intl` + subfolder `/[lang]/`, корректный hreflang ([14](14-seo-strategy.md)) |
| Контент-хаб, правки без разработчика | headless-CMS (catch-reports/блог/видео редактирует клиент/Влад) |
| Лиды → менеджер/CRM, WhatsApp | API-роуты формы → CRM + WhatsApp-deeplink; гео-логика форм ([16](16-payments-legal.md)) |
| Будущее приложение (Модель 2) | Общий API/данные (headless) → веб и приложение на одном бэкенде |

**CMS-кандидаты:** Sanity / Payload / Strapi (headless, мультиязык, медиа). Выбор — в Discovery.

## Почему НЕ WordPress
- Кастомная логика (конфигуратор-презентация, многошаговый лид, гео-формы, будущий букинг) на WP — это борьба с платформой и плагинный зоопарк.
- Производительность/скорость на мобайле и тонкий контроль SEO/GEO/schema хуже, чем у Next.js.
- WP «упрётся» при росте к платформе/приложению — клиент сам это предположил.
- WP оправдан для простой визитки-блога — но это противоречит «мировому бренду» и продающей логике.

## Почему НЕ «отдельное приложение + WP» (ответ на вопрос клиента)
Разрыв данных = **двойная разработка и поддержка** в будущем: контент, лиды, пакеты пришлось бы вести в двух несвязанных системах. При общем headless-бэкенде сайт и будущее приложение используют **одни данные и API** → дешевле в долгую, единый источник правды. Отдельное приложение сейчас вообще не нужно (Модель 1 — лид-сайт); приложение — это Модель 2, и к ней мы готовим архитектуру, а не строим преждевременно.

## Про Laravel (что говорил Адам)
Laravel — валидный кастом-бэкенд, если потребуется тяжёлая серверная логика/букинг-движок (Модель 2). Для Модели 1 (контент + лиды) **headless-CMS + Next API-роуты достаточно и быстрее в запуске**. Решение «когда вводить выделенный бэкенд (Laravel/Node)» — на этапе Модели 2 (букинг/оплаты/B2B-кабинет). Архитектуру держим так, чтобы добавить его без переписывания фронта.

## Дорожная карта стека по фазам
- **Фаза 1 (Флагман):** Next.js + headless-CMS + Vercel; форма→CRM/WhatsApp; SEO/GEO-фундамент (schema, hreflang, sitemap, robots/llms).
- **Фаза 2 (Рост):** аналитика (GA4 + Яндекс.Метрика), A/B, email/CRM-автоматизация, расширение локалей.
- **Фаза 3 / Модель 2:** выделенный бэкенд (Laravel/Node) под букинг/наличие/оплаты, B2B-кабинет, **мобильное приложение** на общем API (React Native/Expo — переиспользует данные и часть логики веба).

## Нефункциональное
- Хостинг/данные с учётом [16](16-payments-legal.md) (РФ-формы → мессенджер/гео-логика; цены в USD).
- Аналитика: GA4 + Яндекс.Метрика (RU).
- Производительность как KPI (Core Web Vitals) — премиальность считывается и через скорость.

> Точные версии/CMS/провайдеры фиксируем в **Discovery** ([17](17-discovery-sprint.md)).
